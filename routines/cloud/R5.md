# R5 — Brain Health Check (cloud)

**Trigger:** Claude Code Routine, Sunday 09:00 (firm timezone).

**Operational state location:** writes `firm/brain-health/YYYY-WW.md` to the **Ledger** per ADR-001. R5 does not mutate any `.xlsx` file. R5 reads from the **Second Brain repo** (the Curator-synced private GitHub repo) and writes its scan report to the Ledger.

## The bridge

Cloud Routines run on Anthropic's infrastructure and cannot reach the user's local my-curator MCP server. R5 uses a different access path: the Curator app syncs its semantic graph two-way to a private GitHub repo (the **Second Brain repo**). R5 clones that repo at every run, scans the markdown pages under `wiki/<firm-curator-domain>/`, and produces the same kind of health report it used to produce via MCP — except via plain file operations instead of the 17 MCP tools.

Trade-off: no semantic-similarity scans (those need the MCP). R5 can detect broken wikilinks, orphan pages, and stale pages — which is most of what the weekly check needs.

> **Migration note (ADR-002, v1.1.0+):** the per-partner `<firm>-secondbrain` repo is retired as a framework primitive. In v1.2.x this prompt still reads the legacy Curator-sync bridge repo; the replacement read-path — the **Firm Brain** repo's `collective/<firm-domain>/wiki/` — becomes the target in v1.3. New installs may point `{{SECOND_BRAIN_REPO_URL}}` at the Firm Brain repo (`<firm>-brain`) instead of provisioning a separate secondbrain repo; the scan logic is identical, only the path prefix differs.

## Prompt body

```
You are running the weekly Second Brain health check for an ØØT organisation.

Two repos are configured as GitHub connectors for this Routine:
  1. The Ledger (read-write, signing key from Step 9 of the installer) — you write the weekly report here.
  2. The Second Brain repo (read-only, fine-grained Contents:Read PAT) — you READ the firm's Curator-synced markdown pages here.

Template variables your installer set:
  {{LEDGER_REPO_URL}}              — e.g. https://github.com/acme/acme-ledger.git
  {{SECOND_BRAIN_REPO_URL}}        — e.g. https://github.com/acme/acme-secondbrain.git
  {{SECOND_BRAIN_SUBFOLDER}}       — e.g. wiki/acme-studio  (scope reads to this domain only)
  {{TODAY}}                        — YYYY-MM-DD
  {{WEEK}}                         — YYYY-WW iso year+week (e.g. 2026-W19)

Your task:

1. Clone both repos into a fresh per-run working directory using the connectors above:
   `WORKDIR=$(mktemp -d)`, then clone the Ledger into `"$WORKDIR/ledger"` and the
   Second Brain repo into `"$WORKDIR/secondbrain"`. Use a unique dir per run (not a
   fixed path) so re-runs never collide on a stale clone.
   (Pattern C, ADR-001: code_execution clones with PATs injected into HTTPS URLs.)

2. Read every `.md` file under {{SECOND_BRAIN_SUBFOLDER}}/ in the Second Brain clone.
   For each page extract:
   - YAML frontmatter (tags, status, slug, links_to)
   - Wikilinks of the form [[slug]] or [[slug|alias]]
   - Last-modified timestamp from git log (`git log -1 --format=%cI -- <path>`)
   - The page's own slug (basename minus .md, or the frontmatter `slug` if present)

3. Compute the four health metrics:

   a. BROKEN WIKILINKS — every [[slug]] in any page where `slug` is not the basename
      of any other page in the same domain. Output: list of (source_page, broken_slug).

   b. ORPHANS — pages that no other page wikilinks to. Exclude pages tagged
      `status: index` or `status: landing` (which are intentionally leaf nodes).
      Output: list of orphan page slugs.

   c. STALE — pages tagged `status: active` (or no status frontmatter) whose
      git-mtime is older than 90 days from {{TODAY}}. Output: list of (slug,
      days_since_modified).

   d. PAGE COUNT — total markdown files scanned, by tag/status group.

4. (Optional, if time permits) Auto-fix typo-correctable broken wikilinks:
   for each broken_slug, find a slug in the same domain within Levenshtein
   distance 2. If exactly one match exists, fix the link in the source page.
   IMPORTANT: this requires write access to the Second Brain repo. R5 has
   READ-ONLY access — so we DON'T auto-fix here. Instead we LIST the typo
   candidates in the report so the user can fix them via the Curator app.

5. Write the report to {{LEDGER_REPO_URL}}'s `firm/brain-health/{{WEEK}}.md`
   following the template at `templates/brain/brain-health-week.md`. Include:
   - Counts for each metric
   - The full lists (broken wikilinks, orphans, stale pages)
   - Suggested typo-fixes (informational only)
   - Total pages scanned
   - Domain scope: {{SECOND_BRAIN_SUBFOLDER}}
   - Sync freshness: how old is the latest commit in the Second Brain clone

6. Stage + signed-commit + push the Ledger:

        cd <ledger-clone> && git add firm/brain-health/{{WEEK}}.md && \\
          git commit -S -m "R5: brain-health snapshot for {{WEEK}}" && \\
          git push origin main

7. Post a summary to Slack #brain-health if any metric is non-zero:
   "Brain health snapshot for {{WEEK}}: {{broken_count}} broken wikilinks,
   {{orphan_count}} orphans, {{stale_count}} stale pages. Full report:
   [link to {{LEDGER_REPO_URL}}/blob/main/firm/brain-health/{{WEEK}}.md]"

Failure handling:
  - Second Brain clone fails (auth, network): retry once after 30s. If still
    failing, write a report flagged "Second Brain unreachable" with the error,
    commit it (so the gap itself is audit-trailed), and post to Slack #ops.
  - Ledger clone or push fails: retry once. If still failing, escalate to #ops.
  - No pages found under {{SECOND_BRAIN_SUBFOLDER}}: write a report flagged
    "domain empty" and post a Slack reminder to ingest documents into the
    firm's Curator domain.
```

## Expected outputs

- Weekly Second Brain health snapshot at `firm/brain-health/YYYY-WW.md` in the Ledger.
- Slack `#brain-health` post if any metric is non-zero.
- (R5 does NOT auto-fix anything. Typo-correctable wikilinks are LISTED in the report; the user fixes them in the Curator app on their next sync-up.)

## Setup checklist

1. Confirm Step 12 of the installer (Second Brain bridge) was completed — `state["second_brain_repo_url"]` should be set in `~/.oot/wizard-state.yaml`.
2. Confirm your Curator app's GitHub sync is enabled and `wiki/<firm-domain>/` has at least a few pages in the synced repo. (Open the Second Brain repo on github.com; you should see your wiki structure mirrored there.)
3. Create the fine-grained PAT with **Contents: Read** on the Second Brain repo (per the wizard's instructions at Step 12).
4. Claude Code → `/schedule` → new Routine, Sunday 09:00.
5. Upload this prompt body.
6. Configure **two** GitHub connectors:
   - **Primary** (Ledger): the repo from `{{LEDGER_REPO_URL}}` with the signing key from Step 9. **Read-write.**
   - **Secondary** (Second Brain): the repo from `{{SECOND_BRAIN_REPO_URL}}` with the fine-grained Contents:Read PAT. **Read-only.**
7. Set template variables `{{LEDGER_REPO_URL}}`, `{{SECOND_BRAIN_REPO_URL}}`, `{{SECOND_BRAIN_SUBFOLDER}}` to the values from your wizard state file.
8. Confirm **code execution is enabled** for this Routine.
9. Configure Slack connector with `#brain-health` + `#ops` channels.
10. Manual test fire on a Saturday. Verify the snapshot lands at `firm/brain-health/YYYY-WW.md` in the Ledger via signed commit.

## Why this is the bridge

The Curator's local-only my-curator MCP server gives partner workstations 17 powerful tools (semantic search, graph traversal, scan_wiki_health, fix_wiki_issue). Cloud Routines can't reach those. The bridge is **the Curator's two-way GitHub sync** — the same feature partners use to keep their Second Brain consistent across multiple machines. R5 piggy-backs on this: it reads the synced markdown files directly. No MCP needed.

What R5 gains: works on cloud track without an always-on machine.
What R5 loses (vs. privacy-track equivalent with local MCP):
- No semantic similarity (`scan_semantic_duplicates`) — defer to Gen-2.
- No auto-fix of typo-correctable wikilinks — R5 can only LIST them. Manual fix via Curator app.
- 24-hour staleness on the synced data (Curator syncs once per day by default; R5's report includes the sync timestamp so this is visible).

Docs: see `docs/AUTOMATION-PIPELINE.md` § "How the bridge works" and `installer/wizard.py` `step_12_secondbrain_sync` for the install-time wiring.

## Privacy-track equivalent

See `routines/privacy/R5.md`. In privacy track the my-curator MCP runs on the same machine as the Routine itself — no bridge needed, full 17-tool access. The privacy R5 keeps the auto-fix behaviour and adds `scan_semantic_duplicates` since the MCP is reachable.
