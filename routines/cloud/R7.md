# R7 — Klarna Test Trigger (cloud)

**Trigger:** GitHub webhook event — PR labelled `ai-replaces-human`. Optionally also: manual trigger from a partner via Slack slash command.

> ⚠️ The auto-labeller (Phase 8) applies the label per `.github/labeler.yml`. The discipline: the partner reviews the auto-label within 24 hours; confirms or removes; the dismissal reason trains the labeller.

> **Schema (ADR-004):** Decision_Log column **M `status`** is the Excel home for the lifecycle state machine — a literal enum (`scoring | remediation | monitoring | proceeded | held`), data-validated, never a formula. It is distinct from column I `decision` (the formula-driven threshold verdict); the two may disagree transiently. R7 writes and updates M directly. The prompt body below reflects the ADR-004 appended-row contract: append A–G literals (including the F/G reviewer columns), write the H/I/K formulas, and write M=`scoring` on create.

## Prompt body

```
You are launching a Klarna Test scoring for an ØØT organisation. The trigger is PR #{{PR_NUMBER}} in repo {{REPO}} labelled 'ai-replaces-human'.

Your task:
1. Generate a test_id following format KT-YYYY-NNN.
2. Append a row to klarna-test.xlsx Decision_Log per the appended-row contract (ADR-004, templates/excel/SPEC.md):
   - A-G literals: test_id (A), today's date (B), decision_summary from PR title + description (C), trigger='pr_label' (D), trigger_ref=PR URL (E), scorer partner_id (F), non_beneficiary_reviewer partner_id (G).
   - Column H (total_score) MUST write the formula =IFERROR(VLOOKUP(A{R},Klarna_Score!A:L,12,FALSE),"").
   - Column I (decision) MUST write the formula =IF(ISBLANK(H{R}),"",IF(H{R}>=14,"PROCEED","HOLD")). Do NOT write a literal.
   - Column K (review_date_90d) MUST write the 90-day review date as a real date cell (step 10).
   - Column M (status) literal = 'scoring'.
   (Pre-ADR-004, R7 appended A-G literals but never wrote H/I/K or the F/G reviewer columns — the Finding-6 bug class. This contract closes it.)
3. Identify the affected partner(s) — partners whose primary function (per their Output Spec history at firm/partners/<id>/output-specs/) overlaps with the PR's automated capability. Cross-reference the PR's diff signatures against Output Spec expected_outcome fields.
4. Identify the non-beneficiary reviewer — a partner whose variable pay or long-tail does not increase as a result of the action. Read X1 Monthly_Variable rolling 90 days; check X2 Long_Tail_Schedule. The affected partner cannot be the non-beneficiary reviewer (Q7 of the rubric).
5. Block PR merge by posting a FAILING GitHub status check named oot/klarna-test against the PR's head SHA. The status check is implemented by .github/workflows/klarna-gate.yml (Phase 8 deliverable). The workflow re-runs on every push and reads klarna-test.xlsx Klarna_Score for the matching test_id; sets the check passing only when the Klarna_Score `total` (column L, =SUM(B2:K2)) >= 14 AND scorer_signoff = Yes AND non_beneficiary_signoff = Yes. (Note: `total_score` is the Decision_Log column H — a VLOOKUP mirror of this `total`; the gate reads the source `total` on Klarna_Score, not the Decision_Log mirror, and never consults the `status` column.) The firm must have configured branch protection requiring this status check; without that protection, the gate is advisory rather than enforcing.
6. Post to Slack #klarna-test: "{{TEST_ID}}: PR #{{PR_NUMBER}} ({{TITLE}}) requires Klarna Test scoring before merge. Affected partner(s): {{LIST}}. Non-beneficiary reviewer: {{ASSIGNED}}. Scoring window: 5 business days. Reference: governance/KLARNA-TEST.md."
7. Email the founder + affected partner(s) + non-beneficiary reviewer with the same content.
8. Open a Ledger page at firm/klarna-tests/{{test_id}}.md following the template at templates/brain/klarna-test-page.md with the full context.
9. Monitor the Klarna_Score sheet for the test_id; on completion, update Decision_Log column M `status` per the ADR-004 lifecycle: 'proceeded' (I=PROCEED → the gate turns green automatically, merge unblocks) or, if I=HOLD, 'remediation' (gaps being fixed for a re-score) or 'held' (decision stands; gate stays red, remediation list posted to #klarna-test). Once a 'proceeded' row enters its 90-day review window set M='monitoring'; close it back to 'proceeded' when the K review date passes and the outcome is recorded.
10. Schedule the 90-day review (Q9 of the rubric) — write the date to Decision_Log column K (real date cell, per step 2), set a calendar reminder.

Failure handling: if scoring isn't complete in 5 business days, escalate to founder for direct decision; never auto-proceed without a complete score.
```

## Expected outputs

- Klarna Test entry in `klarna-test.xlsx` Decision_Log + Klarna_Score.
- PR merge blocked via `oot/klarna-test` status check.
- Slack `#klarna-test` post + emails.
- `firm/klarna-tests/{{test_id}}.md` Ledger page.
- 90-day review scheduled in calendar.

3. **Configure branch protection** to require `oot/klarna-test` status check.
4. Claude Code → `/schedule` → new Routine, GitHub-event trigger (label-applied event).
5. Upload prompt body. Attach `change-management`, `code-qa`, `compensation-attribution` Skill Packs.
6. Configure connectors: GitHub (read + write with signing — for X4 commits + PR status check), Slack, Email.
7. **Confirm code execution is enabled** — required for the X4 openpyxl write.
8. **Plan tier:** R7 fires push you toward Max plan (15/day) regardless of partner count; if Klarna gate is active, do not stay on Pro.
9. Test: open a draft PR, manually apply `ai-replaces-human` label, verify R7 fires and a signed commit lands updating X4 Decision_Log.
