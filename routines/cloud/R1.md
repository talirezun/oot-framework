# R1 — Daily Output Capture (cloud)

**Trigger:** Claude Code Routine, daily 18:00 in the firm's primary timezone.

**Operational state location:** `firm/excel/partner-output-ledger.xlsx` in the firm's Ledger GitHub repo (per ADR-001). Routine clones the Ledger, mutates X1 via openpyxl, signed-commits, pushes. Track-symmetric with the privacy variant.

## Prompt body

```
You are running the daily output capture for an ØØT organisation. Today is {{TODAY}}.

Your task: read all output signals from the past 24 hours across the following sources, and append rows to partner-output-ledger.xlsx (file X1) per its schema in templates/excel/SPEC.md.

Sources to read:
1. GitHub: all commits across all firm repos in the last 24 hours. Include: commit hash, author (note any Co-authored-by: trailers indicating AI assistance), files changed, lines added/deleted, PR association if applicable.
2. Slack channels listed in {{TRACKED_CHANNELS}}: outputs explicitly tagged with #output, contracts mentioned in #commercial, deals closed in #sales.
3. Google Workspace (read-only via the native connector): documents created or significantly modified in {{TRACKED_FOLDERS}}. The native connector can read Sheets, Slides, PDFs, MS Office files; it does NOT write back. All writes happen via Pattern C (Ledger, openpyxl, signed commit) — see "Implementation" below.

For each output:
- Generate a log_id following the format OL-YYYYMMDD-NNN.
- Determine the partner_id from the author/owner (consult the partners Ledger page if uncertain; if multiple authors, create one row per partner with a fractional output_value note).
- Classify output_type per the X1 schema (commit | pr_merged | contract_signed | deal_closed | spec_drafted | review_completed | design_shipped | content_published).
- Reference the output_spec_ref from the Ledger (firm/output-specs/) if one exists; flag for human review if not.
- Estimate value_tier (S/M/L/XS) based on the output_spec value envelope; default to L if no spec exists.
- Estimate ai_authored_pct from commit trailers and change patterns (best-effort; partner can correct).
- Set rework_within_30d to "No" by default. This field is updated retroactively by R1 on every subsequent run within the next 30 days using the deterministic detection rule:

  > A new commit C_new marks an earlier commit C_old as rework=Yes if all four conditions hold: (a) same partner_id on both commits; (b) C_new timestamp is within 30 days after C_old; (c) ≥50% file overlap between the two commits' changed-files sets; (d) C_new's commit message OR associated PR title/description matches the regex \b(fix|revert|hotfix|redo|retry|reapply|backout|rollback)\b (case-insensitive). When all four hold, R1 updates the prior X1 row's rework_within_30d to "Yes" (which zeroes out its computed_variable via the X1 column-L formula). Log every retroactive update in the daily Brain summary under a "## Retroactive rework detections" heading.

- Resolve partner_multiplier (X1 column J) by reading X2's output_multiplier (column F) for that partner at runtime and writing the resolved number into J. Do NOT use a cross-workbook Excel formula.
- Compute value_envelope (column K) per the embedded lookup (XS=100, L=500, M=2000, S=8000).

Append rows to X1 Output_Log. Do NOT edit existing rows except to flip rework_within_30d per the rule above.

Dedupe (makes re-runs idempotent): before appending, read the existing Output_Log and collect every output_ref (column E) from rows whose date (column B) is within the last 45 days. SKIP any candidate whose output_ref is already in that set — it was captured on a prior run. Running R1 twice on the same day (e.g. after an infra retry) must append zero duplicate rows and pay nobody twice.

Implementation (Pattern C, ADR-001):
1. Clone the Ledger into a fresh per-run dir: `WORKDIR=$(mktemp -d) && git clone <LEDGER_REPO_URL> "$WORKDIR/ledger" && cd "$WORKDIR/ledger"`. Use a unique dir per run (not a fixed `/tmp/brain`) so same-day re-runs never collide on a stale clone; a fresh clone is already current, so no `git pull` is needed after it.
2. Open `firm/excel/partner-output-ledger.xlsx` with openpyxl.
3. Find the next empty row using column A (log_id) as the determinant. Do NOT use `ws.max_row + 1` — the embedded value_envelope_table at O1:P5 makes max_row report 5 even when sample data only reaches row 4, which would leave a ghost gap.
3a. Build the dedupe set: read existing Output_Log rows and collect output_ref (column E) from every row whose date (column B) is within the last 45 days. Any candidate whose output_ref is in this set is a duplicate from a prior run — skip it. Log the skipped count in the daily Brain summary. This makes re-runs idempotent.
4. For every row R appended:
   - Write columns A-I directly (log_id, date, partner_id, output_type, output_ref, output_spec_ref, value_tier, ai_authored_pct, rework_within_30d).
   - Resolve column J (`partner_multiplier`) from X2 (`firm/excel/reward-species-declaration.xlsx`) at write time and write the resolved number.
   - **MUST write column K formula** `=VLOOKUP(G{R}, $O$2:$P$5, 2, FALSE)` (with `{R}` substituted to the row number).
   - **MUST write column L formula** `=K{R}*J{R}*IF(I{R}="Yes", 0, 1)` (with `{R}` substituted).
   - Without K and L formulas, the row contributes 0 to Monthly_Variable's SUMIFS — silent zero-pay bug. Do NOT skip these writes.
5. For retroactive rework_within_30d updates, edit the corresponding rows in place.
6. Save X1.
7. Write the daily Brain summary at `firm/output-logs/YYYY-MM-DD.md` per the template at `templates/brain/daily-output-log.md`. Include the count skipped as duplicates.
8. Stage + signed-commit + push: `git add firm/excel/partner-output-ledger.xlsx firm/output-logs/YYYY-MM-DD.md && git commit -S -m "R1: append <N> outputs for {{TODAY}}; <K> retroactive rework updates" && git push origin main`.

Post a short summary to Slack #output-log: "{{TODAY}} captured {{N}} outputs from {{P}} partners. Anomalies: {{LIST_OR_NONE}}."

Failure handling: if any source is unreachable, log the failure to the daily summary, capture from available sources, and post a notice to #ops. If the signed-commit / push fails (signing key unavailable, branch protection rejects, network), retry with exponential backoff up to 3 attempts; if still failing, escalate to #ops — DO NOT downgrade to an unsigned commit silently.
```

## Expected outputs

- Rows appended to `firm/excel/partner-output-ledger.xlsx` in the Ledger.
- Ledger page at `firm/output-logs/YYYY-MM-DD.md`.
- One signed commit on `main` carrying both changes.
- Slack summary post to `#output-log`.

## Setup checklist

1. Open Claude Code → `/schedule`. Or use the web dashboard at [claude.ai/code/routines](https://claude.ai/code/routines), or the Claude Code desktop app's "New Remote Task" feature. Routines run on Anthropic's cloud — your laptop can be closed.
2. Create a new Routine. Trigger: daily 18:00 (firm timezone).
3. Upload the prompt body above.
4. Attach Skill Packs: `compensation-attribution`, `my-curator`, `reporting-business-review`.
5. Configure connectors: GitHub (read **and write** with signing — Ledger cloneable + pushable), Slack (read + post #output-log + #ops), Google Workspace (read-only).
6. **Confirm code execution is enabled** for this Routine (default for Pro+).
7. Set `{{TRACKED_CHANNELS}}` and `{{TRACKED_FOLDERS}}` template variables. Set `LEDGER_REPO_URL` to the firm's Ledger HTTPS URL.
8. **One-time:** the bot identity executing this Routine (e.g. `oot-bot`) must have a GPG or SSH signing key uploaded to GitHub and `git config --global commit.gpgsign true` set. See `governance/SECRETS-POLICY.md`.
9. Run a manual test fire; verify a signed commit lands on `main` adding a row to X1 and the daily Brain summary.
