# R5 — Brain Health Check (privacy)

**Trigger:** OS-native scheduling, Sunday 09:00.

**No bridge needed.** The cloud-track R5 ([`routines/cloud/R5.md`](../cloud/R5.md)) cannot reach the user's local Curator, so it clones a second repo and does a **file-level** scan of the synced markdown — no MCP, no auto-fix, no semantic-duplicate detection. Privacy-track R5 runs on the **same always-on machine** as the my-curator MCP server, so it calls the Curator's tools **directly**. That locality buys three capabilities the cloud track structurally cannot have (see the per-track delta at the bottom).

**Operational state location:** writes `firm/brain-health/{{WEEK}}.md` to the **Ledger** per ADR-001. R5 does not mutate any `.xlsx` file. It reads the firm's Curator domain via MCP and writes the report to the Ledger via a signed commit.

## cron (Linux)

```cron
# R5 brain-health check — Sunday 09:00
0 9 * * 0 cd ~/{{FIRM_SLUG}} && /usr/local/bin/opencode run --model lmstudio/qwen-3-14b-instruct "$(cat ~/oot-framework/routines/privacy/r5.prompt.md)" >> ~/oot-framework/logs/r5.log 2>&1
```

Install: `crontab -e`, paste, save. Verify: `crontab -l`. OpenCode runs headless against the LM Studio server (hosted by the `llmster` daemon); the my-curator MCP server (local stdio) is configured in the firm folder's scoped `opencode.json`, which is why the command `cd`s into `~/{{FIRM_SLUG}}` first.

## Prompt body

This is R5's **own** prompt — privacy R5 does NOT inherit the cloud bridge prompt. It talks to the Curator directly via MCP.

```
You are running the weekly Brain health check for an ØØT organisation on its always-on
machine. The my-curator MCP server is running locally and its full tool set is reachable.
Today is {{TODAY}}; this report covers ISO week {{WEEK}}.

Template variables your installer set:
    {{FIRM_SLUG}}              — e.g. acme-studio (the firm folder under ~/, holds opencode.json)
    {{FIRM_CURATOR_DOMAIN}}    — e.g. acme-studio (the Curator domain slug to scan)
    {{LEDGER_REPO_URL}}        — e.g. https://github.com/acme/acme-ledger.git
    {{TODAY}}                  — YYYY-MM-DD
    {{WEEK}}                   — YYYY-WW iso year+week (e.g. 2026-W19)

Your task:

1. Call scan_wiki_health on the firm's Curator domain {{FIRM_CURATOR_DOMAIN}}. This returns
   categorised structural issues: broken wikilinks (with or without a suggestedTarget),
   orphan pages, cross-folder duplicates, hyphenation variants, missing backlinks, and
   folder-prefix violations. Record the counts and the full per-issue lists.

2. Call scan_semantic_duplicates on {{FIRM_CURATOR_DOMAIN}} to find pages describing the
   same concept under different slugs (e.g. [[rag]] vs [[retrieval-augmented-generation]]).
   This capability is UNAVAILABLE on the cloud track (it needs the local MCP + an LLM call).
   Run it with a bounded max_pairs so cost stays small. Record every candidate duplicate
   pair with its confidence score. Do NOT auto-merge — semantic-duplicate merges are
   destructive (they delete a page and rewrite every link to it); they are LISTED for
   human review, never auto-applied.

3. AUTO-FIX the SAFE issues via fix_wiki_issue — but ONLY the narrowly-safe class:
   a typo-correctable broken wikilink is safe to fix when scan_wiki_health returned it
   WITH a suggestedTarget AND that target is the single existing slug within Levenshtein
   distance ≤ 2 of the broken slug (exactly one candidate — never fix an ambiguous link
   with two or more near-matches). This is the same characterisation the cloud R5 uses to
   DECIDE what is safe; the difference is that privacy R5 can actually APPLY the fix,
   because it has write access to the Curator on this machine.
   For each auto-fix applied, record (source_page, broken_slug, fixed_to_slug).
   LIST — do NOT fix — everything ambiguous: broken links with no suggestedTarget or with
   ≥2 near-matches, orphan pages (human judgement), and all semantic-duplicate candidates.

4. Classify pages as STALE: pages tagged `status: active` (or with no status frontmatter)
   whose last update is older than 90 days from {{TODAY}}.

5. Write the weekly report to the Ledger clone at firm/brain-health/{{WEEK}}.md, following
   the template at templates/brain/brain-health-week.md. Populate the shared fields (broken
   wikilinks, orphans, stale pages, pages scanned, suggested typo-fixes). Privacy R5 adds
   two fields the cloud template leaves empty:
   - a "## Semantic duplicates" section listing the candidate pairs from step 2, and
   - a "fixed" count in the summary — how many typo-correctable broken links R5 auto-fixed
     this run (the cloud template shows these as "suggested only" because cloud R5 can't fix).
   Because the fixes were applied via the Curator (not via the Ledger), the "Suggested
   typo-fixes" section should instead read "Auto-fixed this run" and list what changed.

6. Commit the report to the Ledger with a signed commit (Pattern C, ADR-001):

    WORKDIR=$(mktemp -d)
    git clone {{LEDGER_REPO_URL}} "$WORKDIR/ledger"
    cd "$WORKDIR/ledger"
    mkdir -p firm/brain-health
    # ... write firm/brain-health/{{WEEK}}.md ...
    git add firm/brain-health/{{WEEK}}.md
    git commit -S -m "R5: brain-health snapshot for {{WEEK}}"
    git push origin main

   Use a unique per-run mktemp working directory (never a fixed path) so re-runs never
   collide on a stale clone. A fresh clone is already current — no git pull needed after it.

7. Post a summary to 4thtech dChat #brain-health if any metric is non-zero:
   "Brain health snapshot for {{WEEK}}: {{broken_count}} broken wikilinks
   ({{fixed_count}} auto-fixed), {{orphan_count}} orphans, {{stale_count}} stale pages,
   {{dupe_count}} semantic-duplicate candidates for review. Full report:
   firm/brain-health/{{WEEK}}.md in the Ledger."

Failure handling:
  - my-curator MCP unreachable (server not running, LM Studio host down): retry once after
    30s. If still failing, write a report flagged "Curator unreachable" with the error,
    commit it (so the gap itself is audit-trailed), and post to dChat #ops.
  - Ledger clone or push fails (auth, network, signing key unavailable): retry once. If
    still failing, escalate to dChat #ops. DO NOT downgrade to an unsigned commit.
  - Domain empty (scan returns zero pages): write a report flagged "domain empty" and post
    a dChat reminder to ingest documents into the firm's Curator domain.
```

## Expected outputs

- Weekly Brain health snapshot at `firm/brain-health/{{WEEK}}.md` in the Ledger (signed commit).
- 4thtech dChat `#brain-health` post if any metric is non-zero.
- Typo-correctable broken wikilinks **auto-fixed in the Curator** (unambiguous single-candidate only). Ambiguous links, orphans, and all semantic-duplicate candidates are LISTED for human review, not fixed.

## Setup checklist (privacy)

1. Confirm the my-curator MCP server is running locally and reachable from OpenCode — it is configured (local stdio) in `~/{{FIRM_SLUG}}/opencode.json`, which is why the schedulers `cd` into the firm folder before running.
2. Confirm the firm's Curator domain `{{FIRM_CURATOR_DOMAIN}}` has at least a few pages ingested (`list_domains` / `get_index`).
3. Confirm the signing key (`oot-bot` GPG/SSH, Trezor/YubiKey-backed slot recommended) is configured for the Ledger push, and `gpg-agent` caches the passphrase.
4. Place the prompt body at `~/oot-framework/routines/privacy/r5.prompt.md`.
5. Create `~/oot-framework/logs/`.
6. Install the launchd plist (macOS) or cron entry (Linux).
7. Manual test fire (any time of week); verify the snapshot lands at `firm/brain-health/{{WEEK}}.md` in the Ledger via a signed commit, and that any single-candidate typo links were fixed in the Curator.

## Per-track delta — what privacy R5 does that cloud R5 cannot, and why

The difference is **same-machine locality**. On the privacy track the my-curator MCP server and the Routine run on the same always-on machine, so the Routine reaches the Curator's full tool set directly (stdio). Cloud Routines run on Anthropic's infrastructure with no route to a local MCP, so cloud R5 falls back to a file-level scan of a synced clone. Concretely, privacy R5 gains:

1. **The MCP tools** — `scan_wiki_health` (categorised, with suggested targets), `fix_wiki_issue`, `scan_semantic_duplicates`, and the rest of the 17-tool set. Cloud R5 has only plain file operations (parse frontmatter, walk wikilinks, git-mtime).
2. **Auto-fix** — privacy R5 applies the single-candidate typo-link fixes it detects (via `fix_wiki_issue`), where cloud R5 can only LIST them (it holds read-only access to the synced repo). The *decision* about what counts as safe is identical across tracks (one unambiguous Levenshtein ≤ 2 candidate); only the ability to act differs.
3. **Semantic-duplicate detection** — `scan_semantic_duplicates` surfaces same-concept-different-slug pages. Cloud R5 cannot do this (it needs the local MCP + an LLM call) and defers it entirely.
