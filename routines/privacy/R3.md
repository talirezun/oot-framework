# R3 — Monthly Variable Calc (privacy)

**Trigger:** OS-native scheduling, 1st of month 09:00.

> ⚠️ **Use the largest local model available.** Compensation accuracy is high-stakes; Qwen 3 14B is acceptable but Llama 3.3 70B (or equivalent) is preferred. The framework's authors recommend running R3 on a model class up from the firm's daily-driver model.

## cron (Linux)

```cron
# Each line chains `lms unload --all; lms load … --ttl 900 -y &&` — the robust scheduled-fire pattern (OPENCODE-SETUP.md §h, TTL/JIT trap).
# Monthly calc — 1st of month 09:00
0 9 1 * * cd ~/{{FIRM_SLUG}} && lms unload --all; lms load llama-3.3-70b-instruct --context-length 32768 --ttl 900 -y && /usr/local/bin/opencode run --model lmstudio/llama-3.3-70b-instruct "$(cat ~/oot-framework/routines/privacy/r3.prompt.md)" >> ~/oot-framework/logs/r3.log 2>&1
# Acknowledgement polling — 10:00 on days 2–8 (covers the 5-business-day review window)
0 10 2-8 * * cd ~/{{FIRM_SLUG}} && lms unload --all; lms load llama-3.3-70b-instruct --context-length 32768 --ttl 900 -y && /usr/local/bin/opencode run --model lmstudio/llama-3.3-70b-instruct "$(cat ~/oot-framework/routines/privacy/r3.prompt.md)" >> ~/oot-framework/logs/r3.log 2>&1
```

> Harness = OpenCode (`opencode run`); model server = local LM Studio, served headless (the LM Studio desktop app's own server on macOS, or the standalone `llmster` daemon on a headless Linux box; load the 70B with `lms load llama-3.3-70b-instruct --context-length 32768 --ttl 3600` — the `--context-length` is required, the default 4096 can't hold the agent's system prompt); the `cd ~/{{FIRM_SLUG}}` runner directory carries the scoped `opencode.json` allow-list (the unattended exception). See R1's note and [`OPENCODE-SETUP.md`](../../installer/agent-assisted/OPENCODE-SETUP.md) → "Scheduled / unattended runs".

The cloud track schedules polling as daily 10:00 fires through day +5; the privacy track needs its own polling cron line because OS-native schedulers do not auto-schedule follow-up fires — hence the second `0 10 2-8 * *` entry above. Day 1's 09:00 fire does the calc; the day-2-to-8 10:00 fires do acknowledgement detection (step 8).

## Prompt body

Identical to `routines/cloud/R3.md` with these substitutions:
- Email → 4thtech dMail.
- Slack `#compensation` → 4thtech dChat `#compensation`.
- Excel reads/writes: same openpyxl operation against a local Ledger clone (per ADR-001; no Excel MCP needed for the Routine path).

**No-op window guard.** Each run must self-check before doing work: if today is outside the day+1..day+5 business-day acknowledgement window for the current month's statements, OR if every partner statement is already acknowledged/disputed/escalated, the run must no-op quickly (log "nothing to poll" and exit) rather than re-scan and re-commit. This keeps the extra day-2-to-8 fires cheap and prevents redundant commits on the Ledger.

## Setup checklist (privacy)

1. Verify the larger model (e.g. Llama 3.3 70B) is downloaded in LM Studio (~40 GB).
2. Confirm always-on machine RAM is sufficient (32-64 GB for 70B at 4-bit quantisation).
3. R1 has produced ≥30 days of X1 data.
4. At least one partner has a populated X2 sheet.
5. Place prompt at `~/oot-framework/routines/privacy/r3.prompt.md` (it opens by instructing the agent to read the compensation-attribution SKILL.md). Confirm OpenCode + the `lmstudio` provider point at the 70B model.
6. Install launchd plist or cron entry.
7. Test on a fresh test sheet before first real run.
